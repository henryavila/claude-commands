# as-status Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add the new core skill `as-status` to Atomic Skills, including Portuguese and English prompt bodies, package metadata, README documentation, and automated install/metadata test coverage.

**Architecture:** `as-status` is a prompt-only core skill, so implementation stays within the existing Atomic Skills packaging model: register the skill in `meta/skills.yaml`, document it in `README.md`, add language-specific prompt bodies under `skills/{pt,en}/core/`, and extend tests that validate metadata parsing and generated install output. No runtime installer logic changes are required because the package already discovers core skills from `meta/skills.yaml`.

**Tech Stack:** Markdown prompt files, YAML metadata, Node.js `node:test` + `node:assert`, existing `installSkills()` test harness.

**Spec:** `docs/superpowers/specs/2026-03-29-as-status-design.md`

---

## File Structure

```text
meta/skills.yaml                                 # Add core skill metadata for as-status
README.md                                        # Add as-status to public skill list
skills/pt/core/status.md                         # Portuguese prompt body for as-status
skills/en/core/status.md                         # English prompt body for as-status
tests/install.test.js                            # Verify generated as-status files and updated skill counts
tests/yaml.test.js                               # Verify skills.yaml parsing still works with as-status metadata
docs/superpowers/specs/2026-03-29-as-status-design.md  # Reference spec only, no further edits expected
```

Design notes for implementers:

- `meta/skills.yaml` remains the single source of frontmatter `name` + English `description` for all IDE outputs.
- `skills/{pt,en}/core/status.md` should follow the same body-only pattern as the existing prompt files in those directories.
- `tests/install.test.js` is the critical regression suite because it validates real generated install output, file existence, and total generated file counts.
- `tests/yaml.test.js` should be kept small: only prove that `parse()` still handles `skills.yaml` shape when `as-status` is present.
